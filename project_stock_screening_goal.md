**"우선매수" heuristic backtested and disproven, then removed (2026-07-03):** User asked whether there's a more established version of the persistence+stability idea. Answer: yes, it resembles real technical-analysis concepts (flag/pennant continuation patterns, Minervini's VCP) — but rather than borrow one on faith, built `PersistentStableRule` (`com.myerp.backtest.rules`) — wraps an existing validated rule and only signals when it fires *again* within `persistDays` trading days of a prior firing on the same stock, with price held inside ±`bandPct` of the earlier signal's close the whole time in between. Registered `-persistent` variants of both `volume-surge-breakout-2.0x` and `breakout-20d-high` (10 days / ±3%, matching the exact parameters the live 우선매수 badge had used) and ran them through the existing `/api/backtest/summary`.

**Result: the heuristic doesn't hold up — for breakout-20d-high it's actively worse than doing nothing.** breakout-20d-high-persistent: KOSPI 17.6% (n=176) vs base rule 28.0% vs baseline 23.0%; KOSDAQ 30.2% (n=43) vs base rule 48.2% vs baseline 40.3% — sample sizes are large enough to trust this, and in both markets the "confirmed/repeated" version underperforms *even the random baseline*, not just the base rule. Plausible read: the first occurrence of a breakout is the freshest momentum; by the time it repeats in a tight range, the move has likely already gone stale. volume-surge-breakout-2.0x-persistent showed mixed numbers (22.2%/50.0%) but on n=8-9 — too small to conclude anything.

**Action taken (not just noted):** removed the "우선매수" badge/highlighting entirely — `CandidateController.history()` no longer computes `priceStable`/`priorityBuy`, `CandidateHistoryEntry` dropped those fields, `BacktestService.isPriceStable()` deleted (was only used for the disproven heuristic), and the frontend's warning-tag/highlighted-row styling removed from `Dashboard.vue`. `stillMatchesToday` (plain factual "is this still signaling today") was kept since it's neutral info, not a validated recommendation. **Lesson for this project generally: when a live-only heuristic gets invented ad hoc (not derived from the backtest), backtest it before shipping it as a UI recommendation, the same rigor as the underlying rules — this one looked intuitive and was wrong.**

**Follow-up (2026-07-03) — user corrected the actual hypothesis, retested, still inconclusive/unpromising:** User clarified `PersistentStableRule` tested the wrong thing — they'd asked about a signal firing *once*, then the stock going quiet (±3%) for 10 days, with THAT quiet period itself becoming the entry trigger (classic flag/consolidation-after-breakout pattern), not the base rule firing a second time. **Result: inconclusive, not promising.** Sample sizes are far too small to trust either way — volume-surge-consolidation10: KOSPI 12.5% (n=8), KOSDAQ 0% (n=1); breakout-consolidation10: KOSPI 10.0% (n=20), KOSDAQ 0% (n=2). The compound condition (base signal is already rare across the 300-stock universe, then requiring a subsequent 10-day quiet period on top shrinks the sample drastically). What little data exists is well below baseline (23%/40%), so nothing here supports adopting it, but it's not a confident disproof either the way `-persistent` was (n=176/43) — just insufficient evidence. **Verdict: neither the repeat-signal version nor the quiet-consolidation version of "signal stability implies priority" survived testing. Current best approach remains ranking today's candidates purely by how many of the two validated base rules (volume-surge-breakout-2.0x, breakout-20d-high) match — no persistence/stability-based priority layer on top, at least until a much larger dataset (more history, or a broader/longer-running universe) makes this kind of compound-condition backtest statistically meaningful.**

**Volume profile (매물대) + real-time search added (2026-07-03):** User argued support/resistance/trend concepts are worth adding since they reflect trader psychology (not fundamentals) and requested (1) deriving buy/sell price levels from 매물대 (volume profile) instead of flat percentages, and (2) a real-time intraday search feature for the ERP.

*Volume profile*: `com.myerp.backtest.VolumeProfile` — buckets a price range into 20 bins over a lookback window, attributes each day's full volume to the bin containing that day's CLOSE (approximation — only daily OHLCV available, not true intrabar ticks), flags bins ≥1.3× the window-average as significant support/resistance zones. `VolumeProfileService` builds one for a stock as of a given date (default 60-day lookback). Tested it as a new signal rule too (`VolumeProfileBreakoutRule` — fires when close crosses above the nearest resistance zone computed from data strictly before that day, to avoid look-ahead): KOSPI 24.0% (+1.0%p over baseline, n=6,803), KOSDAQ 43.4% (+3.1%p, n=5,774) — beats baseline but noticeably weaker than the existing `breakout-20d-high` (+5.0%p/+7.9%p), so **not adopted as a third validated rule**, left available in the backtest tool only.

*Buy/sell levels*: `CandidateService.buyLimit()`/`stopLoss()` now use `VolumeProfile.nearestResistanceAbove()`/`nearestSupportBelow()` around the signal close, falling back to the old flat +3%/-5% only when no significant zone exists on that side. Verified real levels now vary per stock (e.g. 아모레퍼시픽 came out +0.65%/-4.5% instead of the old fixed +3%/-5%) — still **not independently backtested as trading triggers**, just more grounded price guidance than an arbitrary flat percentage.

*Real-time search*: New `com.myerp.quote` package. `RealtimeQuoteService` calls KIS's 주식현재가시세 (`FHKST01010100`, `/uapi/domestic-stock/v1/quotations/inquire-price`) directly — bypasses the daily-batch DB entirely, so it reflects the actual current intraday price whenever called (during market hours) rather than yesterday's close. Bonus: KIS also returns its own pivot-point support/resistance (`pvt_frst_dmrs_prc` etc. — a *different*, simpler prior-day-H/L/C-based technique, not volume profile, but useful reference context) — surfaced alongside. `GET /api/quote/search?keyword=` searches the full 4,377-stock master (not just the 300-stock screening universe) by name-substring or code-prefix, common stock only, top 20 by market cap. `GET /api/quote/{itemCd}` returns the live quote. Frontend got a third "종목검색" tab: debounced search box → result list → "현재가 조회" button → card showing price/change/OHLC/volume/pivot levels.

**Caveat repeated because it matters here too**: this "real-time" search still only works while the WSL box happens to be running — same session-bound constraint as everything else (see [[project_wsl_environment_ephemeral]]). The Windows Task Scheduler auto-boot step was still not confirmed done as of this writing; without it, "real-time during market hours" only actually happens if a Claude session or manual WSL terminal is open at that moment.
