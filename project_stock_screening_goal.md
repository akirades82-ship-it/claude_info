**"우선매수" heuristic backtested and disproven, then removed (2026-07-03):** User asked whether there's a more established version of the persistence+stability idea. Answer: yes, it resembles real technical-analysis concepts (flag/pennant continuation patterns, Minervini's VCP) — but rather than borrow one on faith, built `PersistentStableRule` (`com.myerp.backtest.rules`) — wraps an existing validated rule and only signals when it fires *again* within `persistDays` trading days of a prior firing on the same stock, with price held inside ±`bandPct` of the earlier signal's close the whole time in between. Registered `-persistent` variants of both `volume-surge-breakout-2.0x` and `breakout-20d-high` (10 days / ±3%, matching the exact parameters the live 우선매수 badge had used) and ran them through the existing `/api/backtest/summary`.

**Result: the heuristic doesn't hold up — for breakout-20d-high it's actively worse than doing nothing.** breakout-20d-high-persistent: KOSPI 17.6% (n=176) vs base rule 28.0% vs baseline 23.0%; KOSDAQ 30.2% (n=43) vs base rule 48.2% vs baseline 40.3% — sample sizes are large enough to trust this, and in both markets the "confirmed/repeated" version underperforms *even the random baseline*, not just the base rule. Plausible read: the first occurrence of a breakout is the freshest momentum; by the time it repeats in a tight range, the move has likely already gone stale. volume-surge-breakout-2.0x-persistent showed mixed numbers (22.2%/50.0%) but on n=8-9 — too small to conclude anything.

**Action taken (not just noted):** removed the "우선매수" badge/highlighting entirely — `CandidateController.history()` no longer computes `priceStable`/`priorityBuy`, `CandidateHistoryEntry` dropped those fields, `BacktestService.isPriceStable()` deleted (was only used for the disproven heuristic), and the frontend's warning-tag/highlighted-row styling removed from `Dashboard.vue`. `stillMatchesToday` (plain factual "is this still signaling today") was kept since it's neutral info, not a validated recommendation. **Lesson for this project generally: when a live-only heuristic gets invented ad hoc (not derived from the backtest), backtest it before shipping it as a UI recommendation, the same rigor as the underlying rules — this one looked intuitive and was wrong.**

**Follow-up (2026-07-03) — user corrected the actual hypothesis, retested, still inconclusive/unpromising:** User clarified `PersistentStableRule` tested the wrong thing — they'd asked about a signal firing *once*, then the stock going quiet (±3%) for 10 days, with THAT quiet period itself becoming the entry trigger (classic flag/consolidation-after-breakout pattern), not the base rule firing a second time.
